# Burgenton Design System — Specification

**Status: specification only — nothing in this document has been implemented.** This defines the visual language for the Burgenton rebuild, to be applied against the page templates defined in the approved website blueprint and built on top of the architecture decisions in `template-audit.md`.

**Brand color check:** the 85-page source PDF was searched for any color/hex/RGB/CMYK/Pantone/brand-guide reference — none exists. The three colors supplied are therefore treated as authoritative and are not overridden:

| Role | Hex |
|---|---|
| Primary | `#040404` |
| Accent | `#FA0808` |
| Neutral | `#747474` |

Every ramp, pairing, and usage rule below is derived mathematically from these three seed colors (documented with WCAG contrast ratios), not from arbitrary substitutes.

---

## Design Intent

Every decision below is in service of five words: **professional, reliable, quality, trust, engineering excellence.** Concretely, that translates to:

- **Near-black over pure black** as the dominant surface/text color — softer than `#000`, reads as considered rather than default.
- **Red used as a high-signal accent, never as decoration** — one primary action per screen gets it. A palette where everything is red has no hierarchy; a palette where red is rare makes the red mean something (a quality Burgenton's own copy already leans on: warranty, trust, urgency-to-act).
- **A tight, consistent 8pt spacing grid and restrained corner radii** — precision over decoration. Generic templates round everything 12–20px and drop shadows on every card; a precision-equipment brand should feel more engineered than that.
- **Generous whitespace and a disciplined type scale** — the content itself (from the 85-page spec) is dense and benefit-heavy; the layout's job is to make that scannable, not to compete with it.
- **Photography-led, not illustration-led** — real machines, real specs, real numbers (mono numerals for capacities/dimensions) signal engineering credibility better than icons or abstract graphics.

## 1. Color Palette

### 1.1 Core ramps

All ramps are generated by mixing each seed color toward white (tints, lighter numbers) and toward near-black (shades, darker numbers) — not toward pure black/white, so the ramps stay tonally consistent with the brand rather than washing out.

**Primary (near-black) — surfaces, text, chrome**
| Token | Hex | Typical use |
|---|---|---|
| `primary-0` (white) | `#FFFFFF` | page background, card surfaces |
| `primary-50` | `#F0F0F0` | subtle section tint, disabled fills |
| `primary-100` | `#D9D9D9` | dividers, borders on light surfaces |
| `primary-200` | `#B9B9B9` | placeholder text, disabled text |
| `primary-300` | `#969696` | tertiary text on light surfaces |
| `primary-400` | `#6D6D6D` | secondary text on light surfaces |
| `primary-600` | `#2C2C2C` | dark surface variant (cards on dark sections) |
| `primary-700` | `#181818` | footer/dark-band background |
| `primary-900` (base) | `#040404` | body text on light surfaces, primary buttons, header |

**Accent (red) — the one color reserved for meaning**
| Token | Hex | Typical use |
|---|---|---|
| `accent-50` | `#FFF3F3` | error/warning banners, badge backgrounds |
| `accent-100` | `#FEE1E1` | hover background for red text links |
| `accent-300` | `#FD9090` | disabled-state accent |
| `accent-500` (base) | `#FA0808` | large headlines, icons, borders, links on dark backgrounds, hover glows — **not** small body text or default button fills (fails AA at small sizes, see §1.2) |
| `accent-600` | `#DC0808` | **solid button fill** (white text on this passes AA — see §1.2) |
| `accent-700` | `#B50707` | button hover/active state |
| `accent-800` | `#8B0606` | button pressed state, dark-mode accent borders |

**Neutral (gray) — secondary text, backgrounds, borders**
| Token | Hex | Typical use |
|---|---|---|
| `neutral-50` | `#F7F7F7` | page section background (alternating light bands) |
| `neutral-100` | `#EEEEEE` | card background on white pages, input backgrounds |
| `neutral-200` | `#DEDEDE` | borders, dividers |
| `neutral-300` | `#CACACA` | disabled borders, secondary text on dark surfaces |
| `neutral-400` | `#AEAEAE` | secondary text on dark surfaces (AA-safe, see §1.2) |
| `neutral-500` (base) | `#747474` | secondary/supporting body text on white |
| `neutral-600` | `#636363` | captions, metadata |
| `neutral-700` | `#4D4D4D` | tertiary headings on light surfaces |
| `neutral-800` | `#363636` | rarely used — near-primary dark tone |

### 1.2 Verified contrast rules (WCAG AA)

These are not stylistic suggestions — they were checked mathematically so the palette is accessible by construction, closing the exact gap flagged in the template audit (global focus/contrast issues).

| Pairing | Ratio | Passes |
|---|---|---|
| `primary-900` text on white | 20.5:1 | AA normal text ✅ |
| White text on `primary-900` | 20.5:1 | AA normal text ✅ |
| `neutral-500` text on white | 4.67:1 | AA normal text ✅ |
| `accent-500` text/icon on white | 4.12:1 | **AA large text/UI only** — do not use for body copy |
| White text on `accent-500` fill | 4.12:1 | **AA large text/UI only** — do not use for default button fill |
| White text on `accent-600` fill | 5.15:1 | AA normal text ✅ — **use `accent-600` as the real button-fill token** |
| White text on `accent-700` fill | 7.00:1 | AA normal text ✅ — hover/active state |
| `accent-500` on `primary-900` (dark sections) | 4.98:1 | AA normal text ✅ |
| `neutral-400` on `primary-900` (dark sections) | 9.24:1 | AA normal text ✅ |
| `neutral-500` on `primary-900` (dark sections) | 4.39:1 | **fails** — do not use base neutral for text on dark backgrounds, use `neutral-400` or lighter |

**Rule of thumb baked into the system:** `accent-500` (`#FA0808`) is the *brand* red — use it for large display type, icon strokes, thin borders, link color on dark backgrounds, and glows. `accent-600` (`#DC0808`) is the *functional* red — use it wherever red sits behind white button text. This two-step distinction is why the accent ramp has both; treat them as different jobs, not different moods.

### 1.3 Semantic colors

The accent red is reserved for brand/CTA meaning, so status colors are a separate, restrained set — used only in forms and system feedback, never in marketing sections:

| Semantic | Hex | Note |
|---|---|---|
| Success | `#1E8E3E` | form success states only |
| Warning | `#B5750A` | form/inline warnings only |
| Error | `#B50707` (= `accent-700`) | reuses the accent ramp deliberately — an error is still "red," just the deep, AA-safe step |
| Info | `#2C2C2C` (= `primary-600`) | neutral, not blue — keeps the palette closed to 3 families |

No blue is introduced anywhere in the system — a common way generic templates dilute a brand is by defaulting to blue for links/info states. Burgenton's link/info color is near-black or accent, never blue.

### 1.4 Photography note (not a UI color)

The equipment photography itself carries safety-yellow and black livery — that's real and shouldn't be fought in image treatment (see §9), but it is **not** promoted to a UI/chrome color. Yellow appears only inside photographs, never in buttons, text, or backgrounds. This keeps the 3-color brand system intact while still letting the product imagery read as authentically industrial.

## 2. Typography Scale

### 2.1 Typeface pairing

| Role | Typeface | Why |
|---|---|---|
| Headings / display | **Archivo** (700/800/900) | A confident, geometric grotesque used by a number of modern industrial and corporate rebrands (wide, engineered letterforms at large sizes) — reads as "international corporate," not "template default." Variable font, so weight can be tuned per breakpoint without extra files. |
| Body / UI | **Inter** (400/500/600) | The current standard for clean, highly-legible international corporate UI text; enormous language coverage (relevant for a pan-African audience), excellent at small sizes for the benefit-checklist-heavy copy this site is full of. |
| Technical/numeric accents | **IBM Plex Mono** (500) | Reserved narrowly for equipment specs — capacities, dimensions, model numbers, warranty durations ("2-YEAR"), stat counters. A monospace numeral treatment is a small, deliberate signal of engineering precision on spec tables and the trust micro-bar, used nowhere else. |

*(This replaces the audit-reviewed Rubik/Rajdhani pairing — Rajdhani was serviceable but reads closer to "gaming/motorsport HUD" than "premium industrial corporate"; Archivo + Inter is the more deliberate choice for the brand attributes requested.)*

### 2.2 Scale (fluid, `clamp()`-based — replaces the audit's per-breakpoint manual overrides)

| Token | Min (mobile) | Max (desktop) | Weight | Line-height | Use |
|---|---|---|---|---|---|
| `display` | 40px | 76px | Archivo 800 | 1.05 | Hero headline only |
| `h1` | 32px | 56px | Archivo 800 | 1.1 | Page-level heading |
| `h2` | 28px | 42px | Archivo 700 | 1.15 | Section heading |
| `h3` | 22px | 30px | Archivo 700 | 1.2 | Sub-section / card group heading |
| `h4` | 18px | 22px | Archivo 700 | 1.3 | Card title |
| `h5` | 16px | 18px | Archivo 600 | 1.35 | Small heading / eyebrow-adjacent |
| `body-lg` | 17px | 18px | Inter 400 | 1.7 | Lead paragraph under a heading |
| `body` | 15px | 16px | Inter 400 | 1.75 | Default paragraph text |
| `body-sm` | 13px | 14px | Inter 400 | 1.6 | Captions, form hints, metadata |
| `eyebrow` | 12px | 13px | Inter 600, uppercase, `letter-spacing: 0.08em` | 1.2 | The "OUR SERVICES"-style label above headings — a pattern already present in the template (`section-title-one`) and worth formalizing |
| `button` | 14px | 15px | Inter 600, uppercase, `letter-spacing: 0.03em` | 1 | All button labels |
| `mono-spec` | 13px | 14px | IBM Plex Mono 500 | 1.4 | Spec values, stat counters, model numbers |

**Rule:** headings never go below Archivo 700; using 400/500 weight on a heading is a design-system violation (it's what makes a site feel templated rather than engineered).

## 3. Spacing System

An 8pt base grid — every spacing value in the system is a multiple of 8 (with a single 4px half-step for tight inline gaps). This is what the audit flagged as entirely missing from the current template (magic-number padding everywhere); this table is the fix.

| Token | Value | Use |
|---|---|---|
| `space-1` | 4px | icon-to-label gaps, tight inline spacing |
| `space-2` | 8px | form field internal padding, small gaps |
| `space-3` | 12px | button internal vertical padding |
| `space-4` | 16px | card internal padding (mobile), default gap |
| `space-5` | 24px | card internal padding (desktop), grid gutters |
| `space-6` | 32px | gap between related elements (heading → body) |
| `space-8` | 48px | gap between a section's header block and its content |
| `space-10` | 64px | space between stacked components within a section |
| `space-12` | 80px | section vertical padding (mobile) |
| `space-16` | 128px | section vertical padding (desktop) |
| `space-20` | 160px | hero vertical padding (desktop) |

**Section rhythm rule:** every full-width section uses `space-12` top/bottom on mobile and `space-16` on desktop as its outer padding, no exceptions — this single rule is what gives the page a consistent, premium vertical rhythm instead of the current template's section-by-section improvisation.

## 4. Grid

- **Columns:** 12-column grid, carried over from the existing Bootstrap 5 foundation (no reason to replace a working, well-understood grid system).
- **Gutter:** `space-5` (24px) mobile → `space-6` (32px) desktop, replacing the flat 30px Bootstrap default with tokens tied to the spacing scale above.
- **Breakpoints:** keep the audit's existing 6-tier breakpoint set (`<576 / 576–767 / 768–991 / 992–1199 / 1200–1399 / ≥1400`) — it's sound and already matches Bootstrap 5, no need to invent a new one.
- **Container max-widths:** `540 / 720 / 960 / 1140 / 1320 / 1400px` (Bootstrap defaults), with one addition — a **content max-width of `720px`** for long-form text blocks (About narrative, article body copy) so paragraph line length never exceeds ~75 characters regardless of viewport, which the current template has no rule for at all.
- **Product/card grids:** 3-up desktop → 2-up tablet → 1-up mobile as the default for Product/Solution/Support/Article card grids (matches the existing `services-item-two` 3-column pattern from the audit, which is worth keeping structurally).

## 5. Border Radius

A tighter, more "engineered" scale than the template's soft, slightly arbitrary 5px/8px/10px mix — precision over decoration:

| Token | Value | Use |
|---|---|---|
| `radius-none` | 0px | full-bleed images, hero media, trust micro-bar |
| `radius-sm` | 4px | inputs, small tags/badges |
| `radius-md` | 6px | buttons, cards |
| `radius-lg` | 10px | large media panels, modals |
| `radius-full` | 999px | pills, avatar/badge circles, the warranty badge treatment |

**Rule:** nothing in the system uses an arbitrary radius outside this table (closing the audit's "3px/8px/10px with no rationale" finding). Most components live at `radius-md` (6px) — enough to soften without looking soft.

## 6. Button System

One component, defined variants and states — replacing the audit's four undocumented, overlapping button classes (`.common-btn` / `.submit-btn` / `.border-btn` / `.dark-btn`).

**Variants**
| Variant | Fill | Text | Use |
|---|---|---|---|
| Primary | `accent-600` | white | one per view — "Request a Quote," "Submit Application" |
| Secondary | `primary-900` | white | "View Models," "Contact Support" — supporting actions |
| Outline | transparent, `1.5px` `primary-900` border | `primary-900` | on white/light sections, tertiary actions |
| Outline-inverse | transparent, `1.5px` white border | white | on dark/image sections |
| Ghost/text | transparent | `primary-900`, underline on hover | inline text-level actions ("View all articles") |

**Sizing**
| Size | Height | Horizontal padding | Text |
|---|---|---|---|
| `sm` | 36px | `space-4` (16px) | `button` scale, no icon |
| `md` (default) | 48px | `space-5` (24px) | `button` scale + optional trailing icon |
| `lg` | 56px | `space-6` (32px) | hero/CTA-band use only |

**States**
- **Default → Hover:** fill darkens one ramp step (`accent-600`→`accent-700`; `primary-900`→`primary-700`); outline variants invert to filled.
- **Focus-visible:** `2px` solid `accent-500` outline, `2px` offset — this directly fixes the audit's global `outline: none` accessibility failure. Every interactive element in the system inherits this same focus ring, no exceptions.
- **Active/pressed:** fill darkens one further step, `scale(0.98)` transform, 100ms.
- **Disabled:** `neutral-200` fill, `neutral-400` text, no hover/focus behavior, `cursor: not-allowed`.

Icons inside buttons (arrow, plus) sit at `16px`, `space-2` gap from the label, and are the only place a button ever uses a second color — icon inherits text color, never accent-on-accent.

## 7. Cards

A single base card component with content-driven variants, replacing the ad hoc per-section styling in the current template.

**Base anatomy:** `white` surface (or `neutral-50` when placed on a white section, for separation) → optional media slot (16:9 or 4:3, see §9) → `space-5` internal padding → `h4` title → `body-sm` description → optional meta row (icon list) → optional footer (link or button).

**Elevation:** `shadow-sm` at rest (see §14), `shadow-md` + `translateY(-4px)` on hover, 200ms ease-out — consistent across every card variant.

**Variants**
| Variant | What's added to the base | Used for |
|---|---|---|
| Product Card | spec/mono-numeral row, "View Details" ghost button | Product category grids |
| Solution Card | icon badge, "Challenges solved" tag list | Solutions pages |
| Support Card | large icon, single CTA link | Support hub |
| Success-Story Card | company logo, stat callout in `mono-spec`, quote excerpt | Success Stories |
| Article/Resource Card | category eyebrow tag, date, read-more ghost link | Knowledge Centre |
| Stat Card | large `display`-scale mono numeral, label beneath, no border/shadow (sits flush in a stat band) | Home/About stats strip |

**Rule:** card corner radius is always `radius-md` (6px), never mixed with `radius-lg` — mixing radii across card types is one of the fastest ways a layout starts to feel like a generic template.

## 8. Section Layouts

A closed set of section patterns — every page in the sitemap is composed from these, not one-off layouts per page:

1. **Hero** — full-bleed media + overlay, see §9.
2. **Trust micro-bar** — full-width `primary-900` band, 5-item icon+label row (Warranty · Genuine Parts · Technical Support · Dealer Network · Fast Delivery), `space-4` vertical padding, sits directly under hero or header on product/solution pages.
3. **Content + Media split** — 50/50 (desktop) image-and-text pairing, alternates image-left/image-right on repeat use down a page to avoid monotony (About, Solutions, Support all use this repeatedly).
4. **Card grid** — 3-up section for any card variant from §7, with a `section-title-one`-style eyebrow+heading+optional-intro-paragraph header, centered, `max-width: 720px`.
5. **Stat band** — full-width dark (`primary-900`) section, 4-up Stat Cards, Odometer count-up animation on scroll-into-view (reactivating the currently-dormant feature identified in the audit).
6. **Full-width CTA band** — dark or accent-tinted band, single headline + one primary + one ghost/outline-inverse button, used to close Product, Solution, and Support pages.
7. **Logo/partner strip** — grayscale logos on `neutral-50`, color-on-hover, for dealer/customer logos once available.
8. **FAQ/accordion** — for the "Why Choose Burgenton [X]" and "Challenges We Help Solve" content patterns identified in the blueprint as naturally FAQ-schema-eligible.
9. **Form section** — see §12.

## 9. Hero Style

- **Structure:** full-bleed image or video background, `primary-900` gradient overlay (`linear-gradient(180deg, rgba(4,4,4,0.15) 0%, rgba(4,4,4,0.65) 100%)`) — darker at the bottom where text sits, near-transparent at the top so the photograph still reads.
- **Content stack (left-aligned, max-width 640px):** `eyebrow` label → `display` headline → `body-lg` subhead → button pair (Primary + Outline-inverse) → trust-indicator row beneath (feeds directly into the micro-bar pattern in §8).
- **Motion:** headline/subhead/buttons enter with the existing WOW.js `fadeInUp`, staggered 150–200ms apart (kept from the audit — it's the one animation pattern already working well) — but capped so the CTA is interactive within 600ms of load; never gate the CTA behind a longer animation sequence.
- **Height:** `min-height: 640px` desktop / `520px` mobile — tall enough to feel premium, short enough that the fold doesn't hide the first content section entirely on mobile.
- **Category/Solution page heroes** are a shorter variant (`min-height: 360px`) of the same pattern, feeding the breadcrumb directly beneath the overlay text.

## 10. Image Treatments

Two distinct, deliberately separated treatments — mixing them within one module was flagged in the blueprint as a mistake to avoid, and this section makes the rule explicit:

**A. Studio product renders** (white/light background, e.g. the existing concrete pump and forklift photography)
- Displayed on `white` or `neutral-50` card/section backgrounds only — never placed over a dark band or given an overlay.
- No added shadow beyond the card's own `shadow-sm` — the renders already have their own studio lighting/shadow baked in; adding a CSS drop-shadow on top reads as amateurish.
- Consistent padding within their frame (`space-5` minimum) so machines never touch the card edge.

**B. Real-world / site photography** (once genuine African project photos are supplied, per the audit's content gap)
- Used exclusively in Hero, Success Stories, Deliveries, and Site Installations — never mixed into the Product catalog grid.
- Always paired with the gradient-overlay treatment from §9 when carrying text, or a flat `4px` `radius-sm` crop with no overlay when used unaccompanied (gallery grids).
- Subtle `contrast(1.05) saturate(1.05)` filter applied uniformly across all real-world photography, so images sourced from different cameras/lighting conditions still feel like one consistent photographic system rather than a mismatched grab-bag.

**Aspect ratios (fixed, not per-image):** hero `21:9` desktop / `4:5` mobile-crop; card media `4:3`; gallery grid `1:1`; article thumbnails `16:9`.

**Hover (interactive image contexts only — galleries, cards):** `scale(1.04)` on the image within an `overflow:hidden` frame, 300ms ease-out — the frame never moves, only the image inside it, so layout stays stable.

## 11. Iconography

- **Library:** replace the current double-loaded FontAwesome (audit finding — two stylesheets for 20 actually-used icons) with a single **outline icon set** (Phosphor or Lucide — both open-source, tree-shakeable, and available as an SVG sprite so only used icons ship). Outline style, not filled — filled icons read heavier/cheaper; consistent outline strokes read more precise/engineered.
- **Stroke weight:** fixed at `1.5px` across every icon in the system — never mix stroke weights.
- **Size scale:** `16 / 20 / 24 / 32 / 48px` — no arbitrary sizes outside this set.
- **Color:** icons inherit surrounding text color by default (`primary-900` on light, white on dark); `accent-500` is used for icons **only** when the icon itself is the primary visual signal of the module (trust micro-bar icons, "Why Choose Burgenton" checkmarks) — never decoratively.
- **Custom trust-badge set:** the blueprint calls for iconography that doesn't exist in any generic icon library — warranty shield, spare-parts gear, technical-support headset, dealer-network map pin, fast-delivery truck. These five are custom-drawn as a matched set (same stroke weight/corner treatment as the outline library) rather than mixed-and-matched from stock icon packs, since they're the most-repeated visual element on the entire site (trust bar appears on nearly every page per the content spec).
- **Checkmarks** (used constantly in the source content's "Benefits" lists) use a custom circular check-badge at `20px`, `accent-600` fill, white check — not a bare FontAwesome check icon — to give the single most-repeated list-item marker on the site actual brand presence instead of a generic glyph.

## 12. Form Styling

Directly resolves the audit's placeholder-only, unlabeled-field accessibility finding.

- **Field anatomy:** visible `body-sm` label above the field (never placeholder-as-label) → input → optional helper/error text below. Placeholder text, when used, is a supplementary example only ("e.g. +234 800 000 0000"), never the only field identifier.
- **Input style:** `1.5px` `neutral-200` border, `radius-sm` (4px), `white` background, `space-3` internal padding, `body` text.
- **Focus:** border becomes `accent-500`, plus the same `2px` `accent-500` focus-visible outline used on buttons (§6) — one consistent focus treatment across every interactive element in the system.
- **Error state:** border becomes `accent-700` (the AA-safe error red from §1.3), helper text switches to error copy in the same color, small alert-icon prefix.
- **Success state:** border becomes success green (§1.3), used sparingly (e.g. after successful form submission confirmation, not per-field).
- **Checkboxes/radios** (needed for the Partnership Application Form's "Dealer / Distributor / Service Partner" selection and the Contact form's industry/enquiry-type selection): custom-styled square (checkbox) / circular (radio) controls at `20px`, `2px` `neutral-300` border unchecked, `accent-600` fill with white check/dot when checked — never the unstyled browser default.
- **Select fields:** same border/radius/focus treatment as text inputs, with a custom chevron (from the icon set in §11) replacing the native OS arrow for visual consistency.
- **Multi-intent Contact router** (from the blueprint): the four intent options (Quote/Support/Dealer/General) are large tappable segmented-control cards, not a `<select>` — each shows an icon + label, active state uses `accent-600` border + `accent-50` fill, and selecting one conditionally reveals the relevant field set beneath with a simple 200ms height/opacity transition.

## 13. Animation Principles

- **Purpose-first:** every animation either (a) confirms an action happened, (b) directs attention to what changed, or (c) eases a state transition. Nothing animates purely for decoration — this is a B2B lead-gen site where speed-to-CTA matters more than spectacle (carried forward directly from the blueprint's animation guidance).
- **Duration tokens:** `100ms` (micro — button press), `200ms` (state changes — hover, focus, form transitions), `300ms` (content reveals — image zoom, card lift), `600ms` (entrance sequences — hero stagger). Nothing in the system animates slower than 600ms.
- **Easing:** `ease-out` for anything entering or growing (feels responsive), `ease-in` for anything leaving or shrinking, standard `ease` for simple hover color/fill changes. No bounce/elastic easing anywhere — bounce reads playful/consumer, not industrial/corporate.
- **Scroll-triggered entrance:** keep WOW.js `fadeInUp` (audit-confirmed working well) as the single entrance pattern for section headers and card grids — one `translateY(24px)→0` + opacity fade, staggered `100–150ms` per sibling. No new entrance styles are introduced; consistency matters more than variety here.
- **Count-up (Odometer):** reactivated for the Stat band (§8), triggered on scroll-into-view, ~1.5s duration — a controlled exception to the 600ms cap because count-up numbers read as more credible when they take a moment, not instantly snap.
- **`prefers-reduced-motion`:** every entrance/scroll animation is disabled (content simply appears) when the user has this OS-level setting on — a baseline the current template does not implement at all.
- **What's explicitly excluded:** parallax scrolling beyond a very subtle hero Ken-Burns (§9), autoplay video with sound, 3D/WebGL effects, page-transition animations, and any animation that delays the primary CTA becoming interactive.

## 14. Hover Effects

A closed set, applied consistently by component type — not invented per-section:

| Element | Hover behavior |
|---|---|
| Primary/Secondary button | fill darkens one ramp step, 200ms |
| Outline button | fills solid with its outline color, text inverts to white, 200ms |
| Ghost/text link | underline draws in left-to-right, 200ms |
| Card (any variant) | `translateY(-4px)` + shadow elevates from `shadow-sm` to `shadow-md`, 200ms ease-out |
| Card media image | `scale(1.04)` within a clipped frame, 300ms ease-out (see §10) |
| Nav item | `accent-500` underline draws in beneath the label, 200ms |
| Icon-only control (scroll-to-top, search, hamburger) | background fades to `neutral-100` (light) / `primary-700` (dark), 150ms — this also gives these controls a visible, non-color-only hover state, supporting the focus-visible fix in §6 |

## 15. Shadows (Elevation Scale)

Replacing the audit's 10 unrelated one-off shadow values with a 4-level token scale, all neutral (`primary-900` at low opacity, never colored):

| Token | Value | Use |
|---|---|---|
| `shadow-none` | none | flush/full-bleed elements, trust micro-bar |
| `shadow-sm` | `0 1px 3px rgba(4,4,4,0.08)` | resting card state, inputs |
| `shadow-md` | `0 8px 24px rgba(4,4,4,0.12)` | hovered card, dropdown menus |
| `shadow-lg` | `0 20px 48px rgba(4,4,4,0.18)` | modals, mega-menu panel, mobile off-canvas |

No component ever uses a shadow outside this table.

## 16. Background Usage

- **Default page background:** `white`.
- **Alternating section rhythm:** sections alternate `white` → `neutral-50` down the page (no more than 2 consecutive sections share a background) — this is what gives long pages (Product Detail, Solutions, Support) visual rhythm without needing dividers or borders between every section.
- **Dark bands:** `primary-900`, reserved for the Trust micro-bar, Stat band, and CTA bands (§8) — using dark sections sparingly is what makes them land as "important" rather than becoming visual noise.
- **Accent as background:** `accent-600` is used as a full-section background only for the highest-priority CTA moments (e.g. a closing "Ready to build with confidence?" band on Product/Solution pages) — capped at one accent-background section per page maximum, so it retains impact.
- **Texture:** a very subtle (`4–6%` opacity) fine diagonal line or dot-grid texture, `primary-900`-on-`neutral-50`, is permitted as a background detail on dark or stat bands only — a small nod to technical/blueprint drafting texture that reinforces "engineering," applied consistently as a single reusable background asset, not invented per-section.
- **Never:** gradients between the accent red and any other hue (reads consumer/gamer, not corporate), full-bleed accent-color sections behind long body text, or busy photographic backgrounds behind dense paragraph content.

---

## 17. Dark / Light Usage

This system does not implement a user-toggleable dark mode (not a meaningful feature for a B2B equipment-catalog site) — "dark/light" here means **section-level dark and light bands used deliberately for rhythm and hierarchy**, as already scoped in §16:

- **Light sections** (`white`/`neutral-50`) carry the majority of content — product details, solution copy, forms, article content. `primary-900` text on these backgrounds.
- **Dark sections** (`primary-900`) are reserved for: the header/nav (dark header on a light page reads more premium/authoritative than a white header — recommend the header itself sits on `primary-900` with white nav text, red-underline active/hover state), the trust micro-bar, stat bands, footer, and CTA closers. White text, `neutral-400` for secondary/muted text (never base `neutral-500`, which fails contrast on dark per §1.2), `accent-500` for links/icons.
- **The footer is always dark** (`primary-700`, one step lighter than pure `primary-900` for subtle separation from a CTA band directly above it) — consistent with the trust-building "serious company" tone the blueprint calls for; a light, generic footer under a serious industrial brand undersells it.
